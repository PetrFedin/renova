# P3-W33 — Trust + WP1–WP3 (2026-07-19)

Цель: закрыть trust smells и обрывы golden path accept→pay / schedule hub без новых «ещё экранов».

## Сделано

### Trust — реквизиты
- Модель `ContractorProfile.payment_requisites` + alembic `v6w7x8y9z0a1` + sqlite_compat.
- API: `GET /contractors/me/profile`, `POST /contractors/profile`, `GET /projects/{id}/payment-requisites`, portal snapshot поля `contractor_*`.
- Mobile: `buildPaymentRequisites` без hardcoded demo-карт; `PaymentDetailSheet` / `portal` из API; UI сохранения в профиле исполнителя.

### Trust — ЮKassa honesty
- Backend: `demo_allowed()` только `development`/`test`; staging/prod без keys → явная ошибка.
- UI: demo-алерты помечают «без реального списания»; карта недоступна → реквизиты/чек.

### Trust — Kontur
- Webhook `POST /api/v1/esign/webhooks/kontur` → `complete_external_signature` ставит `signed_at` при `status=signed`.

### WP2 — CTA после приёмки
- `WorkAcceptanceScreen`: после accept → Alert «Оплатить» → `budget?tab=payments`.
- Notify `payment_pending` уже ведёт на `/(customer)/(tabs)/budget?tab=payments`.

### WP3 — schedule в hub
- `UnifiedScheduleView`: подрядчик создаёт план-график из этапов в хабе «Сроки» (не orphan `WorkScheduleScreen`).

## Не в этом PR
- Live ЮKassa keys в staging (ops: `YOOKASSA_SHOP_ID` / `YOOKASSA_SECRET_KEY` / webhook secret).
- Полный CO→budget→eSign e2e UI.
- 1С/банк/ФНС live (P4–P5).
